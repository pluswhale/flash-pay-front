# QuickPay — OTC Exchange CRM

Фронтенд для OTC-обменника. Три роли: **Клиент**, **Оператор**, **Администратор**.

---

## Стек

| Слой | Технология |
|---|---|
| UI | React 18 + TypeScript + Tailwind CSS |
| Роутинг | React Router v6 |
| Серверное состояние | TanStack Query (React Query) |
| Real-time | Socket.IO client |
| HTTP | Axios (с автоматическим unwrap envelope + refresh) |
| Архитектура | MVVM: `api/` → `hooks/view-models/` → `components/` → `pages/` |

---

## Быстрый старт

### 1. Требования

- Node.js ≥ 18
- NestJS-бэкенд запущен на `http://localhost:3000`
- В БД создан superadmin-пользователь (номер телефона)

### 2. Установка и запуск

```bash
npm install
npm run dev
# Открыть http://localhost:5173/flash-pay-front/login
```

### 3. Переменные окружения (`.env`)

```bash
VITE_API_URL=http://localhost:3000/api/v1
VITE_WS_URL=http://localhost:3000
```

Если бэкенд на другом порту — поменяйте значения.

## Сценарии использования

### Роль: Администратор

**Вход:**
1. Перейти на `/login`
2. Ввести номер телефона superadmin-аккаунта из базы данных
3. Нажать «Получить код»
4. Посмотреть код в **терминале бэкенда** (в dev-режиме SMS не отправляется реально)
5. Ввести код → нажать «Войти»
6. Автоматический редирект на `/admin/invites`

**Создание инвайт-ссылки для клиента:**
1. На странице «Инвайты» выбрать роль «Клиент», срок действия
2. Нажать «Сгенерировать»
3. Скопировать ссылку вида `http://localhost:5173/flash-pay-front/register?invite=XXXX`
4. Отправить ссылку клиенту

**Создание инвайт-ссылки для оператора:**
1. Аналогично, выбрать роль «Оператор»

---

### Роль: Клиент

**Регистрация по инвайту:**
1. Перейти по ссылке `/register?invite=XXXX`
2. Ввести имя и номер телефона
3. Нажать «Получить код»
4. Посмотреть код в **терминале бэкенда**
5. Ввести код → «Создать аккаунт»
6. Автоматический редирект на `/app/requests`

**Повторный вход:**
1. `/login` → номер телефона → код из терминала → «Войти»

**Создание заявки:**
1. На странице «Мои заявки» нажать «Новая заявка»
2. **Шаг 1 — Направление**: выбрать пару обмена (например, USDT → RUB)
3. **Шаг 2 — Сумма**: ввести сумму, выбрать валюту; опционально — город/страна
4. **Шаг 3 — Способ получения**: выбрать из списка (банкомат, счёт, курьер и т.д.)
5. **Шаг 4 — Детали**: добавить комментарий; проверить сводку → «Сформировать заявку»
6. Автоматический переход в **чат с оператором**

**Работа в чате:**
- Писать свободный текст или использовать кнопки быстрых ответов:
  - «Отправил оплату»
  - «Жду реквизиты»
  - «Получил средства»
  - «Нужна помощь»
- Видеть статус заявки в шапке чата

---

### Роль: Оператор

**Вход:**
1. Перейти по инвайт-ссылке оператора → зарегистрироваться (или `/login` если уже есть аккаунт)
2. После входа — автоматический редирект на `/operator/queue`

**Работа с очередью:**
1. Страница «Очередь заявок» показывает все входящие заявки
2. Новые заявки появляются **мгновенно** без перезагрузки (WebSocket)
3. Можно фильтровать по статусу: Новые / Назначены / В работе / Завершены / Отменены
4. Нажать на заявку → открывается рабочее окно

**Рабочее окно оператора:**
- Левая панель: данные клиента, детали заявки (направление, сумма, способ, комментарий)
- Правая панель: чат с клиентом
- Кнопки действий (согласно машине состояний):
  - Новая → «Взять в работу» (ASSIGNED) или «Отменить» (CANCELLED)
  - Назначена → «Начать обработку» (IN_PROGRESS) или «Отменить»
  - В работе → «Завершить заявку» (COMPLETED) или «Отменить»
- Быстрые ответы в чате:
  - «Принял заявку в работу»
  - «Ожидайте реквизиты»
  - «Перевод отправлен»
  - «Уточните детали»

## Машина состояний заявок

```
NEW ──→ ASSIGNED ──→ IN_PROGRESS ──→ COMPLETED
 └──────────────────────────────────→ CANCELLED
```

Управление переходами только у оператора. Клиент видит текущий статус в чате.

## Полный сценарий тестирования (end-to-end)

1. **Подготовка**: убедитесь, что бэкенд запущен на `localhost:3000` и superadmin создан в БД.

2. **Вход как admin** → `/login` → номер → код из терминала → войти.

3. **Создать инвайт для клиента**: `/admin/invites` → роль «Клиент» → «Сгенерировать» → скопировать URL.

4. **Создать инвайт для оператора**: аналогично, роль «Оператор» → скопировать URL.

5. **Открыть новое окно/инкогнито** → перейти по ссылке клиента → зарегистрироваться → войти автоматически.

6. **Клиент**: создать заявку → выбрать направление / сумму / способ / комментарий → «Сформировать заявку» → открывается чат.

7. **Открыть ещё одно окно** → войти по ссылке оператора → зарегистрироваться.

8. **Оператор**: перейти на `/operator/queue` → новая заявка появится **мгновенно** (WebSocket).

9. **Оператор**: открыть заявку → «Взять в работу» → написать в чате «Принял заявку в работу».

10. **Клиент**: видит сообщение в реальном времени, статус меняется на «Назначена».

11. **Оператор**: «Начать обработку» → переписка → «Завершить заявку».

12. **Клиент**: статус в шапке чата сменится на «Завершена».

## Известные особенности разработки

- **SMS-код**: в dev-режиме код выводится в консоль бэкенда (`[SMS] код: XXXXXX`), реальная SMS не отправляется.
- **accessToken**: хранится только в памяти (не localStorage). После перезагрузки страницы RouteGuard автоматически обновляет его через `/auth/refresh` (refreshToken из localStorage).
- **CORS**: бэкенд должен разрешать `localhost:5173`. Если порт другой — добавьте в `CORS_ORIGINS` на бэкенде.
- **WebSocket**: подключается сразу после успешного входа и используется для real-time обновлений очереди оператора и чата.
